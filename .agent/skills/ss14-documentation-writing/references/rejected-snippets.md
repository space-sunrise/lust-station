# Rejected Snippets (Documentation)

| Зона | Что найдено | Почему отклонено | Сигнал |
|---|---|---|---|
| XML docs | Невалидный закрывающий тег `</summarY>` | Ломает XML-документацию и автогенерацию подсказок | Ошибка тега |
| Language | Doc-комментарии на русском/смешанном языке в коде | Для проектной документации принят английский | Нарушение языкового стандарта |
| Callsite docs | Нестандартный вызов API без пояснения причин (`ignore*` флаги) | Поведение неочевидно при ревью и рефакторе | Скрытый special-case |
| Code comments | TODO-комментарий как замена документации поведения | Это техдолг, а не надежный контракт | `TODO/HACK/FIXME` |
| YAML comments | Длинные многострочные "прологи" в начале файла прототипов | Для прототипов нужна краткость, максимум 1 предложение на пояснение | Стена текста |
| YAML comments | Edit-маркеры (`Start/End`) вместо описания поведения | Не объясняют систему и быстро устаревают | Шумовые метки |
| FTL headings | Заголовок без пробела после `##` (`##bombs`) | Ломает консистентность формата и ухудшает поддержку | Некорректный формат |
| FTL headings | Декоративные разделители (`##########`) | Не дают информации о группе переводов | Шум вместо структуры |
| Code comments | Пошаговый пересказ очевидных операций | Ухудшает читаемость и скрывает реально важные места | Over-commenting |

## Как использовать этот список

1. Не копируй эти фрагменты как "образец стиля".
2. При ревью отмечай такие места как технический риск качества документации.
3. Если нельзя исправить сразу, оставляй узкий TODO в задаче, но не превращай его в постоянную документацию.
