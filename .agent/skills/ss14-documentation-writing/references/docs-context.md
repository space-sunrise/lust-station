# Docs Context (Documentation)

## Роль docs-репозитория

1. Читать `docs` обязательно перед формированием правил.
2. Использовать docs для intent и терминов, а код - как финальный источник поведения.
3. Всегда помнить, что часть страниц может быть устаревшей.

## Ключевые страницы и актуальность

1. `conventions` (обновлено 2026-01-19).
Что брать: требование использовать XML docs (`summary`), подход "комментировать зачем", YAML- и localization-конвенции.
Ограничение: в проекте присутствует legacy-код, не полностью совпадающий с этим стандартом.

2. `fluent-and-localization` (обновлено 2026-01-12).
Что брать: принципы Fluent, структуру ключей и роль FTL как единого слоя пользовательского текста.
Ограничение: страница не является жестким style-guide по частоте и формату групповых комментариев.

3. `forking` (обновлено 2026-02-07).
Что брать: короткие объяснения "почему" при форк-изменениях.
Ограничение: это не руководство по подробной документации подсистем.

4. `yaml-crash-course` (обновлено 2023-09-11).
Что брать: базовый синтаксис YAML и базовую механику комментариев.
Ограничение: документ старый (старше 2 лет), поэтому не использовать как источник строгих проектных правил.

## Практика применения docs

1. Сначала верифицируй правило по свежему коду.
2. Затем уточняй формулировку через docs-страницы.
3. Если docs и код расходятся, приоритет за кодом.
4. Если docs дают общую рекомендацию, но код показывает более строгий рабочий паттерн, фиксируй в skill более строгий вариант.
